\section{Introduction}

meteoearth is a single-page, browser-based globe that visualizes live weather
forecasts from three global models --- NOAA GFS, ECMWF IFS, and ECCC GEM ---
chosen from a model selector. It renders animated wind particles, switchable
scalar overlays (temperature, humidity, pressure, cloud, precipitable water,
wind gusts, precipitation rate, and CAPE), labeled high- and low-pressure
centers, and point readouts on hover and right-click. A time bar scrubs the
forecast from the analysis hour out to two days ahead. It is modeled on the
spirit of \texttt{earth.nullschool.net} but built on deck.gl's
\texttt{GlobeView} with the WeatherLayers GL community edition, and backed by a
small Python pipeline that turns GRIB2 model output into compact PNG textures.

As shown in Figure~\ref{fig:overview}, the workspace is a draggable, zoomable
globe. Wind flows as luminous particle trails; a control panel sits at lower
left; the run timestamp is shown at upper left; and blue \textbf{L} / red
\textbf{H} glyphs mark the centers of the cyclones and anticyclones.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_overview.png}
\caption{The meteoearth globe as it loads: 10\,m wind particles over open ocean
and land, with high/low pressure centers labeled. The control panel (lower left)
selects overlays, wind level, and particle appearance.}
\label{fig:overview}
\end{figure}

\subsection{Design Goals}
\begin{itemize}
  \item \textbf{Live, multi-model data.} A single \texttt{refresh-data.sh} pulls
        the latest cycle of each global model and rebuilds every layer.
  \item \textbf{No build step.} The frontend is one HTML file plus one JS module,
        loading deck.gl and WeatherLayers from a CDN.
  \item \textbf{Compact transport.} Each variable is packed into a PNG and a
        small JSON of metadata, sampled client-side for instant readouts.
  \item \textbf{Reproducible provenance.} Every layer traces back to an explicit
        model field and level (\S\ref{sec:data}).
  \item \textbf{Honest coverage.} A model's overlays reflect exactly what it
        publishes; fields it does not provide are greyed out rather than faked.
\end{itemize}

\section{Installation and Setup}

meteoearth has been developed and tested on macOS (Apple Silicon). Dependencies:
\begin{itemize}
  \item Python 3.10+ with a local \texttt{venv}
  \item A modern Chromium-based browser
  \item (for figures/tests) Selenium with a matching ChromeDriver
\end{itemize}

The repository root contains the operational scripts:
\begin{lstlisting}
./refresh-data.sh        # pull latest cycle of all models + rebuild data/
./refresh-data.sh gfs    # just one model (fast)
./start                  # serve http://localhost:5862 and open the globe
./stop                   # stop the local server
\end{lstlisting}

\texttt{./start} launches a static HTTP server on port 5862 and opens
\url{http://localhost:5862/frontend/index.html}. Run \texttt{./refresh-data.sh}
at least once first so the \texttt{data/} directory is populated. A full refresh
of all three models takes roughly ten to fifteen minutes (most of it fetching
the higher-resolution ECMWF and GEM grids); a single-model refresh is much
faster. The ECMWF client requires the \texttt{ecmwf-opendata} package
(\texttt{pip install ecmwf-opendata}).

\section{Data and Provenance}
\label{sec:data}

meteoearth ingests three global forecast models, each from its own provider:
\begin{itemize}
  \item \textbf{NOAA GFS} (0.5\textdegree{}) --- GRIB2 subsets from the NOMADS
        filter endpoint; cycles 00/06/12/18\,UTC.
  \item \textbf{ECMWF IFS} (0.25\textdegree{}) --- via the official
        \texttt{ecmwf-opendata} client (CC\,BY\,4.0 open data).
  \item \textbf{ECCC GEM / GDPS} (0.15\textdegree{}) --- single-message GRIB2
        files from the Meteorological Service of Canada \texttt{hpfx} server;
        cycles 00/12\,UTC.
\end{itemize}
All three are \emph{global} models, so they fill the entire globe; regional
limited-area models (HRRR, AROME, etc.) are intentionally excluded because they
would cover only a patch. For each model the pipeline downloads the analysis
hour through a two-day forecast (f000--f048, every 3\,hours) at the model's
native resolution.

The variable \emph{names}, labels, units, palettes, and encode ranges live in a
single canonical registry, \texttt{pipeline/variables.py}; how each model fetches
and decodes a given canonical variable lives in \texttt{pipeline/sources.py}. Not
every model publishes every field, so coverage differs by model.
Table~\ref{tab:sources} lists the models actually built and
Table~\ref{tab:coverage} the per-model variable coverage. Neither is
hand-written: both are generated by \texttt{docs/gen\_data\_sources.py} from
\texttt{data/index.json} and the canonical registry, so the manual cannot drift
from what is actually shipped.

\input{data_sources.tex}

For each model, \texttt{pipeline/sources.py} downloads the raw GRIB,
\texttt{pipeline/decode.py} reads it with \texttt{cfgrib} and applies any unit
transform (e.g.\ K\,$\to$\,\textdegree{}C, Pa\,$\to$\,hPa), and
\texttt{pipeline/encode.py} linearly packs the values into an 8-bit PNG (the R/G
channels carry $u$/$v$ for vector fields; a single channel carries scalars)
alongside a JSON of bounds, the encode range, run/valid times, and a color
palette. The whole run is orchestrated by \texttt{pipeline/build.py}, which also
writes \texttt{data/index.json} (the list of sources, each with its run time,
forecast steps, and available variables). The provenance chain for any layer is:
\begin{center}
model GRIB subset $\rightarrow$ \texttt{decode.py} $\rightarrow$
\texttt{encode.py} $\rightarrow$ \texttt{data/<model>/<layer>.f<hhh>.png} +
\texttt{.json} $\rightarrow$ frontend.
\end{center}

\section{Interface Tour}

\subsection{Globe Navigation}
Drag to rotate, scroll to zoom. The view is a true 3-D globe, so layers wrap
seamlessly across the date line and converge correctly at the poles.

\subsection{Control Panel}
The lower-left panel (visible in every figure) carries a title bar with a
\texttt{$-$/$+$} button that minimizes it to a small chip, then several button
rows and four sliders. Every control has a hover tooltip explaining it.
\begin{itemize}
  \item \textbf{model} --- choose the source model (NOAA GFS, ECMWF IFS, ECCC
        GEM); see \S\ref{sec:models}.
  \item \textbf{overlay} --- choose the scalar layer painted on the globe
        (\texttt{none}, temp, RH, MSLP, cloud, PWAT, gust, precip, CAPE).
        Mutually exclusive; overlays the active model lacks are greyed out.
  \item \textbf{wind level} --- switch the particle field between 10\,m surface
        wind and the 250\,hPa jet stream.
  \item \textbf{pressure} --- the \texttt{H\,/\,L centers} toggle (on by
        default); see \S\ref{sec:highlow}.
  \item \textbf{sliders} --- trail length, speed, particle count, and line width
        for the wind animation.
\end{itemize}
A separate \textbf{time bar} runs along the bottom of the screen for scrubbing
the forecast (\S\ref{sec:forecast}).

\subsection{Hover and Pin Readouts}
Moving the cursor over the globe shows a near-cursor badge with the latitude and
longitude and every variable sampled at that point --- including the mean
sea-level pressure. Right-clicking \emph{pins} a point: a persistent panel
(Figure~\ref{fig:pin}) holds the readout until dismissed with the \texttt{\(\times\)}
button or the \texttt{Esc} key.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_pin.png}
\caption{A pinned point over the temperature overlay. The right-hand panel keeps
the full multi-variable readout (wind, temperature, humidity, pressure, cloud,
precipitable water) for the marked location.}
\label{fig:pin}
\end{figure}

\section{Forecast Time Dimension}
\label{sec:forecast}

meteoearth is a forecast, not a single snapshot. For every model the pipeline
ingests the analysis hour and the two-day forecast in 3-hour steps, and the
\textbf{time bar} along the bottom of the screen scrubs through them. The
\texttt{$\triangleright$} button plays the sequence; \texttt{$\langle$} and
\texttt{$\rangle$} step one frame; the slider jumps to any step; and the label
shows the valid time and lead (``analysis'' or ``$+N$\,h''). Steps are loaded
lazily and cached, so the first pass animates as frames arrive and re-scrubbing
is instant. Switching the overlay, wind level, pin, and pressure centers all
follow the selected step. Figure~\ref{fig:forecast} shows the globe at the
$+48$\,h step.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_forecast.png}
\caption{The forecast scrubbed to $+48$\,h. The time bar (bottom center) shows
the valid time and lead; every layer and readout reflects the selected step.}
\label{fig:forecast}
\end{figure}

\section{Multiple Models}
\label{sec:models}

The \textbf{model} row selects which global model supplies the data: NOAA GFS
(0.5\textdegree{}), ECMWF IFS (0.25\textdegree{}), or ECCC GEM (0.15\textdegree{}).
Switching models reloads the current step from that model and updates the run
timestamp at upper left. Because the models publish different fields
(Table~\ref{tab:coverage}), the overlay buttons reconcile to the active model:
fields it does not provide are disabled and greyed out rather than fabricated.
Figure~\ref{fig:models} shows ECMWF selected, where RH, precip, and CAPE are
greyed because ECMWF open data does not publish them, while the finer
0.25\textdegree{} grid is visible in the temperature field.

\begin{figure}[h]

\section{Scalar Overlays}

Selecting an overlay paints a model field on the globe with a per-variable
palette and shows a matching colorbar legend at upper right. Temperature
(Figure~\ref{fig:temp}) runs from deep violet (coldest) to dark red (hottest).
Total precipitable water (Figure~\ref{fig:pwat}) reveals atmospheric rivers as
bright moisture plumes. CAPE (Figure~\ref{fig:cape}) highlights regions of
convective instability --- a proxy for thunderstorm potential. The remaining
overlays --- relative humidity, cloud cover (a grayscale, satellite-like field),
wind gusts, and precipitation rate --- follow the same pattern, and MSLP is shown
later with the pressure centers (Figure~\ref{fig:highlow}).

\section{Wind Particles and the Jet Stream}

The wind field is animated as thousands of particles advected by the vector
texture. The \textbf{wind level} buttons switch the source field between 10\,m
surface wind and 250\,hPa wind. At 250\,hPa the particles trace the jet stream
(Figure~\ref{fig:jet}): fast, coherent ribbons girdling the mid-latitudes. The
sliders tune trail length, speed, particle count, and line width in real time.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_jet.png}
\caption{250\,hPa wind. The particle field resolves the jet stream as
high-speed, narrow ribbons, very different in character from the surface flow.}
\label{fig:jet}
\end{figure}

\section{High and Low Pressure Centers}
\label{sec:highlow}

The \texttt{H\,/\,L centers} overlay labels the eyes of the pressure systems
directly on the globe. It is computed from the mean sea-level pressure field by
WeatherLayers' \texttt{HighLowLayer}, which locates local pressure minima and
maxima and draws a large glyph at each: a blue \textbf{L} for a low-pressure
system (cyclone) and a red \textbf{H} for a high-pressure system (anticyclone),
with the central pressure printed beneath. A search radius of roughly 1500\,km
keeps the labels to the major synoptic systems rather than every minor ripple.
Figure~\ref{fig:highlow} shows the glyphs over the MSLP overlay, where each
\textbf{L} sits at the heart of a visible cyclonic vortex in the wind.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_highlow.png}
\caption{High/low pressure centers over the MSLP overlay. Blue \textbf{L} glyphs
mark cyclones (e.g.\ the 986\,hPa low in the North Atlantic); red \textbf{H}
glyphs mark anticyclones. The central pressure is printed under each letter.}
\label{fig:highlow}
\end{figure}

Hovering a glyph replaces the point readout with a focused summary of that
system: its type, central pressure, and coordinates
(Figure~\ref{fig:highlowhover}). The glyphs ride above any active overlay, so
they can be combined with temperature, moisture, or a bare wind field. The
overlay is on by default and can be toggled off from the control panel.

\begin{figure}[h]
\centering
\includegraphics[width=0.92\textwidth]{fig_highlow_hover.png}
\caption{Hovering the North Atlantic low surfaces its details:
``Low-pressure system, central pressure 986\,hPa, 56.6\textdegree{}\,N,
24.5\textdegree{}\,W.''}
\label{fig:highlowhover}
\end{figure}

\section{Architecture}

\subsection{Pipeline (Python)}
\begin{itemize}
  \item \texttt{variables.py} --- the canonical registry of layers: labels,
        units, kind, encode ranges, unit transforms, and palettes (source
        independent).
  \item \texttt{sources.py} --- one adapter per model (GFS, ECMWF, GEM): cycle
        discovery, how to fetch each canonical variable, and which it supports.
  \item \texttt{decode.py} --- generic GRIB2 $\rightarrow$ numpy via cfgrib,
        handling one- or two-file vector fields.
  \item \texttt{encode.py} --- numpy $\rightarrow$ PNG (byte-packed) plus the
        metadata JSON, via \texttt{encode\_field}.
  \item \texttt{build.py} --- orchestrates fetch/decode/encode for each model and
        forecast hour into \texttt{data/<model>/}, and writes
        \texttt{data/index.json}.
\end{itemize}

\subsection{Frontend (deck.gl + WeatherLayers)}
\texttt{frontend/src/main.js} reads \texttt{data/index.json}, then lazily loads
each forecast step's PNGs and JSON for the active model (cached per
model/step). It builds the deck.gl layer stack: an invisible pickable globe
polygon (so hover fires everywhere on the sphere), the optional scalar
\texttt{RasterLayer}, a coastline \texttt{GeoJsonLayer}, the WeatherLayers
\texttt{ParticleLayer} for wind, the \texttt{HighLowLayer} for pressure centers,
and a pin marker. Switching model or step swaps the in-memory textures and
rebuilds the stack; point readouts are computed in JavaScript by sampling the PNG
bytes and un-scaling them with each layer's \texttt{imageUnscale} range. Hover
and pin require an actual pick on the globe, so they stay silent off the sphere.

\section{Testing and Figures}

All screenshots and figures are produced with Selenium so they are reproducible
and never hand-captured:
\begin{itemize}
  \item \texttt{tests/auto/test\_ui.py} --- end-to-end smoke test (canvas,
        overlays, hover, pin, wind level, legend).
  \item \texttt{tests/auto/test\_highlow.py} --- verifies the pressure-center
        overlay: detection counts, glyph hover badge, toggle behavior, and that
        the point readout still carries pressure.
  \item \texttt{tests/auto/test\_timebar.py} --- verifies the forecast time bar:
        step count, scrubbing, and that readouts change between steps.
  \item \texttt{tests/auto/test\_layers.py} --- verifies the gust, precip, and
        CAPE overlays render with the right legends and readouts.
  \item \texttt{tests/auto/test\_models.py} --- verifies the model selector:
        run-time updates and that unavailable overlays are disabled per model.
  \item \texttt{docs/capture\_figures.py} --- drives the app through the states
        used for the figures in this manual and writes \texttt{docs/figures/}.
  \item \texttt{docs/gen\_data\_sources.py} --- regenerates
        Tables~\ref{tab:sources}--\ref{tab:coverage} from \texttt{data/index.json}
        and the canonical registry.
\end{itemize}

The whole document, including refreshed figures and the data table, is rebuilt by
\texttt{docs/build.sh}.
